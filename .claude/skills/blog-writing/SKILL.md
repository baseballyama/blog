# ブログ記事の文体・構成ルール

このブログの記事を書く・直すときは、以下のルールをすべて適用する。

## 文体

- 説明調のだ・である調。既存記事(posts/ 配下)のトーンに合わせる
- **小説的な書き方は禁止**: 情景描写のフック、ドラマ調の展開、比喩の多用、体言止めの導入
- **能動態を使う**(Google テクニカルライティングガイドに従う)。受動態は動作主が不明・無関係な場合のみ許容
  - NG: 「エラーが報告される」「差分が文書化されている」「共有されるとは限らない」
  - OK: 「エラーを報告する」「差分ごとに原因を文書化している」「共有するとは限らない」
  - 「走らせられる」のような使役 + 可能の重ね形も避け、「動かせる」のような簡潔な形にする
- **AI 的な慣用表現を使わない**: 「効く/効かない」「時間稼ぎ」「壁に当たる」「刺さる」など
  - 代わりに直接的な記述: 「ダメな理由」「問題の発生を遅らせるだけ」「同じ制約に達する」
- **ダッシュによる挿入は日英とも使わない**(「——」「—」「–」すべて禁止)。丸括弧・カンマ・文の分割で処理する。英語の数値範囲はハイフン(46.9-64.4)を使う
- **英語(en.md)は平易に書く**。著者は英語ネイティブではないため、短い文・一般的な語彙を使い、凝った構文や難しい語(nominalization、堅い学術語)を避ける
- 列挙は「1 つ目は〜である。2 つ目は〜である」形式
- 冒頭は自分の状況説明(例: 「私が働いている Flyle では〜」)から入り、「〜を整理する」で本題を提示する

## 主張と根拠

- 評価語より測定可能な事実を優先する。「速い」ではなく数値、「大きい」ではなく実測値
- 数値には必ず出典を付ける。公称値は「公称」と明記、経験談は主語を明示(「私の経験でも」)
- 大きな数字(2 桁の倍率など)は測定条件を先に説明する。エンジン処理と CLI 体験(エンドツーエンド)を区別する
- 「完全」「一切」「ドロップイン」のような強い言葉は、目標と現状を分離して使う。現状の制約・不都合な結果も正面から書く
- 反論されうる主張には条件を付ける(「同期的に待つループでは」など)。予測を事実のように書かない
- ベンチマークは再現条件(マシン・バージョン・回数・中央値/平均・スクリプトの所在)を明記する

## 表記

- 会社名は "Flyle"(先頭大文字)。URL 内の flyle.io 等はそのまま
- 肩書きは ja「VPoT」/ en「VP of Technology」
- フロントマターの title は引用符で囲まない(パーサーは YAML ではなく最初のコロンで分割する自前実装のため、引用符が表示に混入する)

## 構成

- フロントマター(title / date)+ 導入 + tl;dr + 番号付きセクション + まとめ + Flyle 採用 CTA
- en.md / ja.md の 2 言語で posts/YYYYMMDD-slug/ に配置
- CTA 定型: 「私が VPoT を務める [株式会社フライル](https://herp.careers/v1/flyle) では、現在ソフトウェアエンジニアを募集しています。…」
